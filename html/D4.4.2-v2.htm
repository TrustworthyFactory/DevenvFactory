1.   Introduction

1.1. Purpose of the document

The purpose of this document is to provide the user guides for the first prototype of Trustworthy Application factory.

The Trustworthy Application factory is implemented through 3 prototypes:

         The trustworthy java applications development factory,

         The trustworthy web applications development factory,

         The certification tool.

1.2. Intended audience

The intended audience are globally the developers and the certifiers.

In the OPTET environment, the interested actors are:

         The software designers/developers of WP8 participating in the development of new applications for the use cases. In this document, they will find the user guide for the development platforms.

         The OPTET evaluator who has to evaluate a trustworthy application. In this document, he will find the user guide for the certification tool.

1.3. Place within WP4 and within wider OPTET context

In the context of WP4, this document is strongly linked with the D4.3.1 describing the different prototypes. This document provides the user guides.

In OPTET context, the document will be mainly used by the people involved in the two use cases and having applications to be developed.

1.4. Document structure

The remainder of this document is structured as follows: In section ‎2 of the user guide of the trustworthy java applications development factory is given for the first version of the prototype.

In Section ‎3, the user guide of the trustworthy web applications development factory is given as a reference to the OPA website, this first version is identical to the standard OPA environment.

In Section ‎4, the user guide of the certification tool is given for the first version of the prototype.

And in section 6, we give a short conclusion and the next steps for the future versions.

3.1.2. The Authentication phase

When the user wants to develop a new component using the Trustworthy Java applications development factory, he starts the factory and enters his credentials in order to be authenticated. During the start-up of the factory based on eclipse[3], the following view appears in order to enter credentials:

In this view, the login and the password must be entered. Then press OK to validate. If the credentials are valid, the user reaches the workspace development view. If not, an error message appears:

In this version, the login password for the development factory is dev/dev.

 

3.1.3. The Configuration of the environment

The Optet environment needs to be configured in order to work perfectly. The configuration is done under a unique entry point into the Preferences of Eclipse.

To access to the Optet Preferences, click on Windows>Preferences>Optet Main Preferences

The following page appears:

This page is the mother page presenting the Optet project. Under this page, all the Optet configuration pages are located, that is to say:

-          Optet Certificate Preferences

-          Optet SVN Preferences

-          ...

For the developer, only the Optet SVN preferences need to be used.

3.1.3.1  Optet SVN Preferences

This page allows the configuration of the SVN repository used to store the Trustworthiness project coming from the architecture phase.

The required data is:

-          The SVN Repository: the URL of the repository used shall be entered

-          The user credentials: the developer enters the SVN credentials used to interact with the SVN repository. (The credentials can be different from the factory credentials).

 

The Java Trustworthiness factory used a COST as SVN repository. The management of this COTS is not part of the configuration of the factory and the certification tool

 

3.1.4. Development of a Java application

The development of a trustworthy Java application is base on the same principle as a standard Java application. The main differences are the

-          The necessity to link this java development with trustworthy requirement coming from the secure repository

-          The necessity to check the code using the different Optet plugin based on the required trustworthy attributes required for the application.

 

3.1.4.1  The trustworthy requirement

The trustworthy requirement for the Java development are present into the secure repository configure previously. This secure repository is composed by a set of projects. Each project is link to a dedicated java application which must be developed with the factory. The structure of each project is the following and integrate of data coming from the design phase (The TW profile[1], the CSM (Concret System Model)[1], etc...). All these data need to be set into the Secure repository during the design phase under a specific tree:

o   An output directory which contains, at the end, the delivery and the TW certificate;

o   The sources directory which contains the sources produced during the development phase;

o   The TWProfile.xml file which is the file containing the evidence required by the assets [2];

o   The CSM.xml file which contains design data of the component to be developed;

o   The readme.txt file which contains information about the project required by the developer and /or the certifier.

All these files are mandatory in order to start development.

 

3.1.4.2  The Java development step

The developer has three possibilities for a development using the Optet environment:

-          Start a new development from scratch;

-          Recover a previous development already developed using Optet

-          Convert a standard java project into an Optet development.

3.1.4.3  Start an empty SVN project

In order to create a new Optet project, the developer must use the Optet Wizard. For this, click on File>New>Other>Optet certification>Optet Wizard

At this stage, the two next views will be the same as a standard Java project.

The developer must enter the name of the project, the location, the JRE used, etc...

The second view is used to configure the Java settings

The last view links the future development with the SVN Optet project:

With this view, the developer is connected to the secure SVN repository containing the entire trustworthiness project. The developer selects the project he wants to develop. Using the list on the left, the developer browses through the entire created project and sees the description and the content of the SVN repository. In this example, just the CSM.xml, the TWProfile.xml and the readme.txt file are present.

In this example, the checkout option shall not be set because the developer start with an empty source folder.

At the end, the project is created and is ready to be updated by the developer.

3.1.4.4  From an existing SVN project

In this example, the developer is in charge of following the development of an application already stored in the secure SVN repository. In this case, the principle is the same as previously except the phase of checkout. When the developer selects the project to be extracted, he selects the checkout option. At the end of the project, the sources of the development will be checked-out into the project workspace.

3.1.4.5  From an existing development

The last example is when a development has already been done by the developer but not attached to an Optet project. In this case, the developer must click on the Eclipse Toolbar optetMenu>Select project. The following view appears in order to give the opportunity to the developer to select the Optet project:

When the project is selected, the developer validates his choice and selects the Java project he wants to associate. For this, the next view appears:

The developer selects the project and validates the choice. After this, the current Java project is associated with the SVN Optet project defined in the secure SVN repository.

3.1.5. The Analysis phase

During the development of the component, the user can run, at anytime, the analysis (static and runtime) in order to evaluate the quality of his code with respect to the predefined trustworthiness requirements selected in the secure SVN repository.

In order to launch the analysis project, the developer must use the contextual menu and select the desired analysis:

3.1.5.1  The static analysis

When the developer selects the static analysis, all the sources are checked regarding the trustworthiness requirements associated with the project. As a result, all the malformations detected into the code are displayed. The display of this is realized into the OptetAudit view as follows:

In this view, the developer can see:

-          The tool which detects the problem;

-          The file involved;

-          The line where the problem is;

-          The severity of the problem;

-          The rule set;

-          The message of the problem;

-          The recommendation in order to solve it.

 

In this first version, the static analysis tools used are:

-          PMD[5]

-          FindBugs[6]

-          Codepro[7]

-          CheckStyle[8]

 

3.1.5.2  The runtime analysis

When the developer selects the runtime analysis, the result is the execution of the unit test present in the project in order to compute some metrics like the coverage. The data is displayed using the result of the JUnit[9] execution:

For the coverage part, the results regarding the number of classes, methods, lines, etc... are displayed in another view

This display is realised in the OptetTestCoverage view:

 

3.1.6. The Optet Metric computation phase

During this phase, a full analysis (static and runtime) is realised and compute regarding the required TWAttribute needed for this development. The output of the different analysis are compute using the method defined in the D4.3.2 (To resume, the different evidence are weight with predefine values in order to calculate the TWAttribute value).

The result of this computation is display in a dedicated view like this :

the value could be transform in percentage value.

For this, he selects the commit source entry present into the contextual menu. Using this command, the commit is realised in the sources directory of the Optet project (in the secure SVN repository associated).

4.  User guide for certifiers

4.1. Global scenario description

A possible scenario for the certification tool is the following:

-          The certifier starts his certification tools. During this step, the user enters his credentials in order to be admitted into the environment;

-          The certifier configures the environment using the preference pages. During this phase, he configures the access to the SVN repository and the certificate used to signed the DTWC;

-          The certifier creates a new Optet project. He uses the Optet wizard to configure this new project. At this stage, the certifier must select a project with the sources (coming from the development phase);

-          On the project, the certifier opens the workflow view of the certification process:

o   The first step enables to define the software description regarding the input of the SVN file (TWProfile and CSM);

o   The second phase enables to define the trustworthiness problem definition;

o   The third phase enables to define the trustworthiness property specification;

o   The fourth phase enables to run the analysis using the analysis plug-in (using the automatic functionality);

o   The last phase enables to generate the DTWC[2] and the component.

-          When the certificate and the component are created, the certifier saves them in the secure repository.

4.1.1. The Authentication phase

The authentication phase is the same as for the developer part. Please refer to this part

The login password is cert/cert in order to access the certification tool

4.1.2. The Configuration of the environment

The certifier must configure some parameter into the certification tool, like the secure repository, the certificate used, the URL of the certificate repository

4.1.2.1  Optet SVN Preferences

The SVN preferences are the same as the developer part. Please refer to this part.

4.1.2.2  Optet Certificate Preferences

The Certificate Preferences allows the certifier to specify some required data about the generation of the certificate.

 

 

 

At the end of the certification process, the certificate generated by the automatic process is signed using the certificate of the certifier. For that, the certifier must enter the element require in order to use his certificate ((see the keytool part) . This certificate is stored in a key store and the access to this key store is specified (the path, the alias of the certificate, the password of the certificate, the password of the key store).

 

When the certificate is generated, its stored locally and also pushed to external service (web service) in charge to store all the certificates

 

4.1.3. The Selection of the project to certified

The certifier needs to select a project from the secure SVN repository in order to certify it. In this case, the recovery of this project is the same as for the developer part (the case From an existing SVN project). This procedure is the same for all the kind of project (Java, Web, mobile)

 

4.1.4. The certification process

To certify a component, a dashboard is dedicated in order to guide the certifier in the process. This dashboard is called using the toolbar entry OptetMenu>Dashboard process View.

The following view appears:

For each step, a dedicated view helps the certifier to enter the required data. All the data is required for the successful completion of the certificate generation.

4.1.4.1  The software description

This view helps the certifier to describe the system and to define the certification perimeter:

 

In this view, from the root element, we can create a subcomponent and associate either another components, or an attribute or a stakeholder to this component.

-          The Component: a component could be associated with another component, some attribute and some stakeholder.

-          The attributes: For the attributes, the certifier may select their type as InputParameter or OutputParameter and choose if this attribute is selected for the evaluation;

-          The Stakeholder: For the stakeholder, the certifier may select the type as End-user, System or Service. All the defined stakeholders are displayed in the stakeholders list for reuse if needed.

4.1.4.2  The Trustworthiness problem definition

In this second phase, the certifier associates each asset present in the TOE with the potential problems to which the asset might be exposed.

For the selected asset, the associated problems checked by the certifier will be also published in the DTWC. (The list of the possible problems is not already defined).

4.1.4.3  The Trustworthiness property specification

In the next step, the certifier associates each asset present in the TOE with a list of TWAttributes defined in the system:

For this first version the possible TWAttributes are:

-          Complexity (With the metric : Complexity);

-          Dependability (With the metrics : Maintainability and reliability);

-          Security (With the metric : Security).

On the right side, all the TW property specification created by the certifier appears. These TWproperties can be deleted if there is a need to.

4.1.4.4  The evidence

At this stage, the certifier checks the code in order to find evidence to fill the data related to the trustworthiness attributes defined for the system in the evaluation report. The certifier has two possibilities:

-          Manual evaluation: In this case, the certifier must enter manually all the metrics values required by the certification process

-          Automatic evaluation: In this case, the certifier run automatically the static and runtime analysis in order to extract evidences. The computation of all the evidences found will use to fulfill the required TWAttributes metrics

4.1.4.4.1             Manual evaluation

When the certifier selects manual evaluation, the following view appears:

 

For each TWAttribute and the associated metrics, the user sets a value and the mehod used to find this value. The possible methods are:

-          Compute: if the certifier uses an external tool to evaluate the metric;

-          Inspection: if the certifier explores all the files manually in order to evaluate the metric;

-          Native: if the language properties used to develop the application imposes the metric.

The certifier must set all the values required.

4.1.4.4.2             Automatic evaluation

In this case, the same view appears but with some unmodifiable values. The values are coming from the evaluation realised when the certifier clicks on this Automatic button.

When the values are coming from the automatic evaluation, the certifier cant modify them. But, if the automatic evaluation cant fulfill some attribute due to a missing information, the certifier must entre the value manually.

4.1.4.5  The DTWC[2]

This last step is carried out when all the previous steps are done. During this phase:

-          The product is compiled

The compilation of the product is realised using the following interface

Using this interface, the certifier can select 3 possibilities to build the product: Maven, Ant and a Shell script. These three kind of build are the common way to build a product into eclipse. Then, the certifier must select the build file using the browse button and indicate the target option define into the build file.

When the build is finished, the certifier must select the compiled product.

This product will be the element pushed by the technology provider to the marketplace.

 

-          The DTWC is generated using the data collected during the certification workflow, the hash of the product is inserted into the certificate and signed with the certifiers certificate defined in the preference pages of the IDE;

-          The DTWC is transfer to the web service responsible to host the certifications files.

-          The commit of the certificate and the product into the secure repository. This commit is realised automatically without any intervension.

 

 

 

 

5.  User guide for technology provider

5.1. Global scenario description

A possible scenario for the technology provider tool is the following:

-          The technology provider starts his technology provider tools. During this step, the user enters his credentials in order to be admitted into the environment;

-          The technology provider configures the environment using the preference pages. During this phase, he configures the access to the SVN repository and the certificate used to signed the delivery package;

-          The technology provider creates a new Optet project. He uses the Optet wizard to configure this new project. At this stage, the technology provider must select a project with the sources (coming from the certification phase);

-          On the project, the technology provider must create the description file which describes the component.

-          When the component description and the component are created, the technology provider sends them to the trustworthy markeplace.

5.1.1. The Authentication phase

The authentication phase is the same as for the developer part. Please refer to this part

The login password is prod/prod in order to access the technology provider tool

5.1.2. The Configuration of the environment

The technology provider must configure some parameter into the technology provider tool, like the secure repository, the certificate used, the URL of the trustworthy marketplace

5.1.2.1  Optet SVN Preferences

The SVN preferences are the same as the developer part. Please refer to this part.

5.1.2.2  Optet Delivery Preferences

The Delivery Preferences allows the technology provider to specify some required data about the generation of the descrption file and the URL of the default Marketplace used by the technology provider.

 

 

The marketplace URL specified into this preference page is the default Marketplace URL used by the technology provider. In case of this URL will be different, the technology provider could modified it during the exportation phase of the Marketplace.

In the keytool part, the technology provider must provide information in order to configure the access to his certificate. This certification will be used to signed the component description file product by the technology provider on the certified component.

 

5.1.3. The Selection of the project to certified

The technology provider needs to select a project from the secure SVN repository in order to export it into the marketplace. In this case, the recovery of this project is the same as for the developer part (the case From an existing SVN project). This procedure is the same for all the kind of project (Java, Web, mobile)

 

5.1.4. The component description

The description of the component is based on the USDL (Unified Service Description Language) specification. The USDL is a platform-neutral language for describing services. Using the HMI provided by the Technology provider tool, the technology provider is able to create easily this document.

In order to access to this HMI, the shortcut is available into the contextual menu of the project selected

 

The HMI used describe the product is composed of 4 tabs:

-          Description of the business entity

-          Description of the service

-          Technical description of the service

-          The service offering description

 

5.1.4.1  Description of the business entity

This tab allows the technology provider to describe the business entity which develops the software component. The needed information are the name of the company, the legal form, a short description and the URL web site of the company.

5.1.4.2  Description of the service

The second tab is the description of the service himself. The required information are the name of the service, the description, an abstract that resume the service, the web site URL of the service, the optet certificate URL, the creation and modification date.

For the certificate URL, the location used is the location of the certificate into the web service which hosts all the certificates. During the certification process, the certificate was push on this service. The same service must expose all the certificates in order to be consult by the marketplace and the different user.

 

 

5.1.4.3  Technical description of the service

The technical description required the following information : the license of the service, the execution platform of the service, the language used to create the service, a list of dependencies (others services, versions, etc...), the hardware requirement (CPU, RAM memory, Disk memory, etc...) and the version of the service.

 

5.1.4.4  The service offering description

This tab is dedicated to the service offering of the service. We can describe the different prices of the service dependent of the usage condition.

After a description of the offering and the validity time, the offering is composed by one or more Price plan which can contains one or more price component.

For example, we can imagine a price plan for student, which is composed by one pricecomponent with a single price. Then, we can have another pricePlan for industrial client, which could have different price component depending of the number of instance deployed.

 

5.1.5. The delivery to the marketplace

The last phase is to provide the product and the USDL description to the marketplace. For this step, the technology provider must used contextual menu to open the dedicated view

 

The following view appears and the different field must be fulfilled.

The required information is the Marketplace where the component but be send. The default URL is the URL defined into the preference page of the technology provider. This URL could be changed.

The User name/password are not used for the moment

The USDL file must be indicated by the technology provider using the browse button.

The Product file must be indicated by the technology provider using the browse button. The selected file must be in line with the certificate indicated into the USDL file. Some concistency and security check will be realised by the Marketplace to be sur le certificate, the USDL and the product file are the good one.
